# 技术验证计划

## 目的

首个演示的目的是验证技术和制作方向，而非交付完整的叙事片段。

该演示应证明项目能够支持：

- HD-2D 视觉构成
- 在3D空间中用像素角色进行探索
- 物体和NPC交互
- 实时战斗
- 可维护的Godot架构
- 现实的资源管线

## 阶段0：项目基线

目标：建立稳定的 Godot 项目基础。

当前范围：

- 使用 Forward+ 的 Godot 4 项目
- 主场景
- 基础3D环境
- 正交倾斜相机
- 玩家场景
- Sprite3D 玩家占位符
- 基础移动
- 初始文件夹结构

验收标准：

- 项目在 Godot 中可正常打开。
- 主场景按 F5 可运行。
- 玩家可以在 3D 场景中移动。
- 相机提供接近 HD-2D 的倾斜角度。
- 项目可以在不替换基线的情况下进行扩展。

状态：部分完成。

## 阶段1：HD-2D 视觉验证

目标：在投入大型系统或最终美术之前，证明视觉目标的可行性。

范围：

- 为小型测试环境改进3D布局
- 使用 Sprite3D 或 AnimatedSprite3D 渲染像素角色
- 正交相机调优
- 相机跟随行为
- 方向光设置
- 雾效、辉光及基础后处理
- 像素纹理导入设置
- 基础阴影和接地方案
- 使用3D物体进行遮挡测试
- 角色与环境之间的比例测试

验收标准：

- 截图能清晰识别为3D场景中的像素角色。
- 相机角度接近 HD-2D，而非普通的俯视2D。
- 角色看起来接地且比例正确。
- 玩家可移动，无视觉抖动或视角混乱。
- 附近的3D物体可以遮挡或框住角色，而不破坏可读性。

## 阶段2：探索与交互验证

目标：验证基础的 RPG 探索循环。

范围：

- 可复用的可交互组件
- 玩家交互检测
- "按E"风格的提示
- 简单的信息/对话框
- 玩家移动的交互锁定/解锁
- 测试物体，如标志牌、灯、NPC、拾取物或触发区域
- 共享同一系统的小型交互类型集合

验收标准：

- 玩家可接近物体并收到提示。
- 按下交互键可触发正确的响应。
- 交互UI需要焦点时，玩家移动被禁用。
- 交互干净地结束，控制权返回给玩家。
- 系统应支持 NPC、道具、拾取物和事件触发器。

## 阶段3：实时战斗核心验证

目标：验证项目能够支持实时动作RPG战斗。

初始战斗循环：

1. 敌人检测到玩家。
2. 敌人进入战斗状态。
3. 玩家攻击或闪避。
4. 敌人追踪并攻击。
5. 双方均可受到伤害。
6. 敌人可被击败。
7. 战斗结束，恢复探索。

范围：

- 玩家HP
- 敌人HP
- 普通攻击
- 命中框和受击框组件
- 伤害数据
- 击退
- 无敌帧
- 闪避或冲刺
- 具有待机、追踪、攻击和返回状态的敌人AI
- 探索/战斗状态切换
- 用于HP和敌人状态的基础UI

验收标准：

- 玩家可以伤害敌人。
- 敌人可以伤害玩家。
- 攻击具有可辨识的前摇、判定和后摇阶段。
- 受击反应可见。
- 击退不会破坏碰撞或相机可读性。
- 敌人可被击败。
- 战斗结束后游戏回到探索状态。

## 阶段4：战斗表现验证

目标：在 HD-2D 风格中让战斗可读且令人满意。

范围：

- 攻击特效占位符
- 命中火花占位符
- 受伤闪烁
- 地面阴影
- 方向性精灵或面向行为
- 小型相机震动
- 敌人预警指示器
- HP条
- 占位音效
- 粒子测试

验收标准：

- 玩家和敌人的动作在倾斜相机下可读。
- 攻击范围可以理解。
- 命中反馈清晰。
- 特效不会过度遮挡角色。
- 相机风格支持战斗而非妨碍战斗。

## 阶段5：集成技术切片

目标：将视觉、探索和战斗系统整合为一个可玩的小型演示。

可能的流程：

1. 玩家在一个小型车站风格的测试环境中生成。
2. 玩家探索并与一个或多个道具交互。
3. 一个触发或交互事件生成敌人。
4. 玩家进入实时战斗。
5. 玩家击败敌人。
6. 场景改变状态或显示简单的完成信息。

验收标准：

- 视觉风格对原型来说足够连贯。
- 探索和战斗能无缝衔接，无明显错误。
- 玩家能清楚理解何时在探索、何时在战斗。
- 系统之间有足够的分离以便扩展。
- 在开发机器上性能稳定。

## 首次验证优先级

优先级顺序：

1. HD-2D 视觉可读性
2. 玩家移动和相机手感
3. 交互循环
4. 实时战斗循环
5. 战斗反馈
6. 资源管线的可重复性

如果后续功能威胁到更高的优先级，则更高的优先级优先。
